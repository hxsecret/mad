将不可绕过的系统调用与动态采集的高维 Python 语义上下文协同建模，可以弥补系统行为与模型意图之间的语义鸿沟，实现既可拦截、又可解释的动态检测。


## Insight的具体描述
<!-- description: 用简洁的语言描述你的核心Insight。例如："我们发现传统入侵检测系统忽略了攻击行为的时序关联特性，导致对高级持续性威胁(APT)的检测率低下。通过建模攻击行为的时序演化规律，可以显著提升对复杂攻击链的识别能力。"描述应包含：Insight的核心观点、与传统认知的差异、该洞察的价值所在。 -->
在模型供应链攻击场景中，底层系统调用具有难以绕过的优势，能够捕捉文件读写、进程创建、网络连接等关键行为，相对难以被攻击者完全规避；但同一类系统调用序列可能对应完全不同的语义例如缓存加载和数据窃取，因此单独使用系统调用会带来语义歧义与高误报风险。

相反，Python 语义上下文更接近“模型意图”，但如果缺少底层行为证据，攻击者可通过混淆与间接调用绕开高层特征。动态插桩可采集诸如调用栈、关键库 API 语义、张量/配置对象流转、模型加载路径与运行阶段等高维上下文，用于解释模型层面的意图。但攻击者可通过伪装，例如间接调用、延迟触发等方式让高层特征难以稳定覆盖。

因此我们提出“基于语义上下文与底层系统调用协同的恶意模型混合防御系统”，通过时空关联把模型层面的语义与底层系统调用的信息结合：底层负责提供强约束证据并触发拦截，高层负责提供业务语义解释并降低误报，从而在攻击者试图伪装成正常推理/训练流程时仍能暴露其异常意图。

### 第一类解决方案
<!-- description: 概述第一类方法的思路和局限。例如："基于签名的入侵检测方法通过匹配已知攻击特征进行检测，核心假设是攻击具有可识别的模式。局限包括：(1)零日攻击问题：无法检测未知攻击；(2)签名维护成本：需要持续更新攻击特征库；(3)变种攻击漏检：简单变种即可绕过签名匹配。" -->
基于模型文件特征扫描机制在模型被加载执行前，通过对文件结构和字节码指令进行静态解析来进行防御。以Hugging Face 等平台的 Picklescan 检测工具为例，其核心原理是提取出所有用于加载外部依赖的 GLOBAL 操作码，然后将提取出的模块导入列表与预先定义的黑名单（如 os.system、subprocess.run 或 eval 等危险函数）进行比对。因此，静态扫描依赖对危险函数名，然而攻击者可以通过导入看似完全无害的内置函数并在反序列化时动态拼装属性名来逃逸检测。另一方面，AI 模型文件通常并非单纯的 Pickle 文件，而这类特征扫描机制往往只能处理特定的文件格式。

### 第二类解决方案

基于模型反编译的静态污点分析方法，通过对模型文件进行反编译与逆向工程，提取其抽象语法树或控制流/数据流图，进而实施静态污点分析。核心思路是将模型加载时的不可信输入或特定对象标记为Source，并静态追踪这些污点数据是否会沿着执行路径流入危险的系统级调用Sink。由于在静态反编译阶段无法获取模型运行时的真实语义上下文，如具体的变量赋值、环境配置参数、以及框架内部的动态状态，容易将正常的业务错误地标记为攻击路径，从而产生海量误报。另一方面，攻击者通常不会在代码中直接暴露危险的数据流，而是采用伪装的手法，使用看似无害的API组合进行攻击，基于反编译的方法无法识别这类代码层面的伪装。

## 存在的共性技术局限
<!-- description: 分析现有各类方法中存在的共性技术瓶颈。例如："现有方法的共性局限在于：(1)攻击行为孤立检测：将每个告警作为独立事件处理，忽略攻击阶段间的关联；(2)时序信息缺失：未建模攻击行为的时间演化特性；(3)上下文信息不足：缺乏对网络环境、资产价值的综合分析。" -->
现有方法的共性局限在于：（1）缺乏真实的运行时上下文，产生高漏报或高误报的问题。（2）都依赖高度动态的Python语言构建，这使得攻击者拥有极大的操作空间来规避静态检测。（3）只能在 Python 字节码层或高层逻辑层进行浅层解析，一旦恶意数据流或控制流通过反序列化过程被直接灌入底层复杂的扩展库，就能够直接绕过这些防御机制。
