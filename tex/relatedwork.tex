\section{Related Work}

\noindent \textbf{Malicious Code Attacks in the AI Supply Chain.}
Insecure serialization attacks arise from a fundamental trade-off between flexibility and security. To preserve complex model architectures alongside learned weights, frameworks rely on serialization mechanisms supporting arbitrary object instantiation~\cite{NEURIPS2019_bdbca288}, blurring the boundary between passive data and executable code~\cite{3368680}. Consequently, loading a compromised artifact triggers automatic payload execution before any ML computation, and these attacks remain widespread on public hubs~\cite{3765037,casey2024largescaleexploitinstrumentationstudy}. Studies further highlight the systemic risks of such supply chain compromises~\cite{MLSYS2025_75bb91b9}.

Framework-native attacks represent a more recent and stealthier threat. Frameworks like TensorFlow offer lower-level APIs for distributed training and debugging that possess file system access and network communication capabilities~\cite{199317}. Attackers hijack these operations to exfiltrate local data or execute unauthorized commands without introducing obvious malicious syntax. Zhu et al.~\cite{zhu2025} introduced TensorAbuse, embedding persistent attacks directly into the computational graph via TensorFlow APIs. Using officially supported operations, these attacks bypass traditional static scanners. However, these attacks remain dormant during loading and trigger only during inference as tensors flow through the graph~\cite{schulz2025shadowlogicbackdoorswhiteboxllm}, creating a severe semantic gap where security monitors lacking application-layer context cannot distinguish legitimate operations from malicious ones.

\noindent \textbf{Defenses Against AI Supply Chain Attacks.}
Current defenses for AI model supply chain attacks primarily rely on static analysis and signature-based scanning. Fickling~\cite{fickling} decompiles pickle bytecodes into readable Python code to identify embedded malicious payloads. Picklescanning~\cite{picklescanning}, integrated with ClamAV~\cite{clamav}, inspects import tables within pickle files for known malware signatures. ModelScan~\cite{modelscan} extends this approach to broader model formats. However, these tools inspect artifacts for predefined patterns without analyzing dynamic execution~\cite{ohayon2026safepicklerobustgenericml}, failing to establish semantic context between API calls and their underlying intent. 

Beyond these scanning methods, the TensorAbuse framework includes a tool to statically classify API capabilities~\cite{zhu2025}. While this represents progress toward semantic-aware detection, it still operates at a static level and struggles with the ambiguity of API usage, limiting its ability to distinguish benign from malicious intents in practice~\cite{3735968}.

System call analysis has proven effective for malware detection~\cite{102526,101007}, yet have not been specifically adapted to AI model supply chain threats. API2Vec++~\cite{10586831} uses temporal and attribute graph embeddings to characterize contextual behaviors across processes. API-MalDetect~\cite{103704} applies CNNs and BiGRUs to API sequences for deep semantic feature extraction. CruParamer~\cite{9715123} incorporates parameter sensitivity into API embeddings.
